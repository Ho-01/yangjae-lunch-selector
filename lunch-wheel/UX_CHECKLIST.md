# 점심룰렛 UX 검증 규칙

새 기능을 추가하거나 UI를 수정할 때 아래 순서로 확인한다. 기준은 Nielsen의
사용성 휴리스틱, WCAG 2.2 AA, 모바일 터치 관행을 이 서비스에 맞게 축약한 것이다.

## 1. 핵심 흐름

- 첫 방문자가 설명 없이 `모드 선택 → 후보 확인 → 돌리기 → 결과 확인`을 수행할 수 있는가?
- 현재 모드, 로딩, 저장, 오류, 완료 상태가 화면에 즉시 표시되는가?
- 자주 쓰는 행동은 한 화면에서 찾을 수 있고, 관리 기능은 핵심 행동을 방해하지 않는가?
- 빈 목록, 전체 제외, 위치 권한 거절, 네트워크 실패에도 다음 행동을 안내하는가?

## 2. 오류 예방과 복구

- 삭제·투표 마감처럼 되돌리기 어려운 행동은 대상과 결과를 미리 알리는가?
- 비활성 버튼은 왜 사용할 수 없는지 인접 문구 또는 접근 가능한 이름으로 설명하는가?
- 입력 오류는 해당 입력 가까이에 표시하고, 사용자가 입력한 값은 보존하는가?
- API 호출, 저장 버튼은 처리 중 중복 실행되지 않는가?

## 3. 접근성

- 모든 기능을 Tab, Shift+Tab, Enter, Space만으로 사용할 수 있는가?
- 키보드 포커스가 항상 눈에 보이고 화면 밖으로 가려지지 않는가?
- 아이콘 전용 버튼에 `aria-label`이 있는가?
- 폼 입력에 연결된 `label`이 있고 placeholder만 라벨로 사용하지 않는가?
- 상태 변화는 `role="status"`, 치명적 오류는 `role="alert"`로 전달되는가?
- 색만으로 선택, 제외, 성공, 오류를 구분하지 않는가?
- 확대 200%와 `prefers-reduced-motion`에서도 기능을 사용할 수 있는가?

## 4. 모바일

- 320px 너비에서 가로 스크롤과 잘린 텍스트가 없는가?
- 주요 터치 대상은 최소 44×44px이며 서로 너무 붙어 있지 않은가?
- 룰렛과 주요 버튼이 320, 375, 390, 430px에서 겹치지 않는가?
- 소프트 키보드가 입력 필드와 저장 버튼을 가리지 않는가?
- 상·하단 안전 영역과 `100dvh`가 반영되는가?
- sticky 요소가 결과나 마지막 목록 항목을 가리지 않는가?

## 5. 콘텐츠와 일관성

- 버튼은 동사형(`불러오기`, `저장`, `다시 확인`)으로 작성하는가?
- 같은 행동은 같은 색, 크기, 위치, 용어를 사용하는가?
- 제목 계층이 `h1 → h2` 순서를 지키는가?
- 내부 구현 용어(API, 캐시 등)는 사용자의 판단에 필요할 때만 노출하는가?
- 날짜, 거리, 별점, 로딩 문구의 표기 방식이 일관적인가?

## 릴리스 전 테스트 시나리오

1. 팀 모드에서 메뉴 1개 제외 후 룰렛을 돌린다.
2. 모든 메뉴를 제외하고 복구 안내를 확인한다.
3. 내 주변 모드에서 위치 허용·거절을 각각 확인한다.
4. 점심방을 만들고 참여, 준비, 마감, 룰렛 완료까지 진행한다.
5. 메뉴 관리 모달을 키보드만으로 열고 입력·저장·닫기 한다.
6. 320×568, 390×844, 768×1024, 1440×900에서 확인한다.
7. 네트워크를 끊고 오류 메시지와 재시도 가능 여부를 확인한다.

## 통과 기준

- 핵심 흐름을 막는 문제, 키보드 사용 불가, 콘텐츠 손실, 가로 스크롤은 릴리스 차단.
- 작은 정렬 차이와 장식 문제는 후속 개선 가능.
- 새 기능은 위 시나리오를 깨지 않고, 사용자가 새 개념을 추가로 학습해야 할 만큼
  복잡도를 높이지 않을 때만 추가한다.
